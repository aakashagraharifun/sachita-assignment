01

What is Full Stack Development?

Full stack development just means you know how to build a website end to end, instead of only handling one piece of it. You've got the front-end, which is everything the user sees and clicks on, and then the back-end, which runs on the server and handles all the behind-the-scenes work like saving data and processing requests.

When someone calls themselves a "full stack developer", they're saying they can work on both sides without being totally lost on either one. They can build the actual UI, and also set up the server and the database it talks to. Most job ads I've looked at mention a specific stack, like MERN (MongoDB, Express, React, Node.js) or LAMP (Linux, Apache, MySQL, PHP). My course has been focused on MERN so far, so that's the one I'm most comfortable with.

The front-end is where you use HTML for the page structure, CSS for the styling, and JavaScript for anything interactive. So if you click a button and a menu slides open, that's JavaScript doing its job. For larger applications, most companies use a framework like React or Angular because writing a huge app in plain JS gets messy really quickly.

The back-end is where things get a bit more involved. It handles logins, sending and receiving data, saving information, and pretty much anything that needs to remember stuff between visits. You also write APIs, which are basically URLs your front-end can hit to grab or send data. So when you submit a form on a website, the back-end picks up that data, checks it's valid, and stores it in something like MySQL or MongoDB.

02

HTML: The Skeleton

HTML stands for HyperText Markup Language, and it's the foundation of pretty much every website out there. It's not really a programming language though, you can't write logic with it or do calculations. What it does is let you structure your content using tags. Want a heading? Use <h1>. Want a paragraph? Use <p>. That sort of thing.

A big concept I picked up from this module was semantic HTML. In the older days people would just wrap everything in <div> tags, but HTML5 brought in proper tags like <header>, <nav>, <main>, <article> and <footer>. They don't just make your code look neater, they actually tell the browser (and search engines) what each part of the page is meant to be doing.

Why does that matter? Couple of reasons. Accessibility is probably the most important one. People who rely on screen readers need those semantic tags so their software can read the page out in the right order. The other big reason is SEO (Search Engine Optimization). Google looks at these tags to figure out what your page is about, and that ends up affecting how high your site shows up in search results.

HTML can also do a lot more than just hold text on a page. You can build forms where users type stuff in using <input> and <form>, embed videos with <video>, or display info in tables. All these elements together make up what's called the DOM, or Document Object Model. It's a tree of all your page elements, and that's the structure CSS and JavaScript hook into when they style or change anything.

03

CSS: The Aesthetic

CSS stands for Cascading Style Sheets. If HTML is the skeleton of the page, then CSS is what dresses it up. Without CSS, every site on the internet would just look like plain black text on a white background with bright blue links everywhere, which gets old fast. CSS is in charge of all the colours, fonts, spacing, sizing, and layout choices.

The whole idea behind CSS is to keep the design separate from the content. Your HTML holds the actual text and structure, and your CSS controls how it all looks on screen. This separation is genuinely handy in practice. You can give a site a complete visual overhaul just by swapping out the stylesheet, without touching a single line of HTML.

Two layout tools I've been using a lot are Flexbox and CSS Grid. Flexbox is great when you're lining things up in a single direction, like a row of links in a navbar. Grid is better suited for a full page layout that needs both rows and columns at the same time. For responsive design we use media queries, which let the layout change depending on whether someone's viewing the site on a phone, a tablet, or a desktop.

One thing I've grown to like is CSS variables (sometimes called custom properties). You define a colour or a value once at the top of your stylesheet and then reuse it wherever you need. If you ever want to change the theme later, you just update it in one place and everything else follows. CSS also supports hover effects and small animations, which lets you add a bit of feedback or movement to the page without writing any JavaScript at all.

04

Bootstrap Framework

Bootstrap is a CSS framework that was originally built by some developers over at Twitter. The idea behind it is pretty simple. Instead of writing all your CSS from scratch every single time, Bootstrap gives you a bunch of pre-made classes you can just drop into your HTML. It saves a lot of time, especially when you're trying to put something together in a hurry.

The most well-known feature is probably the grid system. It divides the page into 12 columns and lets you decide how many each element should take up. So if you want two boxes side by side that are equal width, you just give each one col-md-6 (since 6 plus 6 makes 12). The cool thing is, it's responsive by default. On a phone screen those boxes stack on top of each other automatically, you don't have to write any extra CSS.

Bootstrap also ships with a ton of ready-made components like navbars, buttons, cards, modals (pop-ups), dropdown menus, and more. You don't really have to style any of them yourself. Just drop in the right class names and they look pretty good straight out of the box. They also behave consistently across different browsers, which is one of those small things that can take ages to fix if you're doing it manually.

In Bootstrap 5 they added a whole heap of utility classes for things like margins, padding, text alignment and colours. So instead of writing a separate CSS rule just to add a little padding, you can stick something like p-3 right onto the element. Some developers aren't a fan of this approach because it clutters up your HTML a bit, but for prototyping it's super quick. Bootstrap also uses Sass under the hood, so if you want to customise the default colours or fonts you can do that as well.

05

JavaScript: The Behavior

JavaScript is what makes a website do things. HTML gives you the structure, CSS makes it look nice, but without JavaScript nothing on the page would respond to you when you click or type. It's a full programming language, you can write functions, use loops, declare variables, all of that. And every modern browser has a JavaScript engine built right in, so the code just runs.

The main job of JavaScript on a webpage is to manipulate the DOM. The DOM is basically the browser's version of your HTML, represented as a tree of objects. With JavaScript you can grab any element on the page and change it, whether that means updating the text in a paragraph, hiding a div, swapping a button's colour when someone clicks it, or adding new elements that weren't there before. It also listens for events, so when a user clicks, types, or submits a form, your code can react to it.

One concept that honestly took me a while to wrap my head around is asynchronous JavaScript. Sometimes you have to fetch data from a server (like pulling posts from an API), and you really don't want your whole page to freeze while it waits for that response. JavaScript handles this using something called Promises, and there's a cleaner syntax called async/await that makes the code much easier to read. To make those server requests, we usually use the Fetch API. This is how modern sites can load new content without doing a full page refresh every time.

What's really nice about JavaScript is that it isn't just a browser thing anymore. With Node.js, you can run JavaScript on a server too, which means you get to use the same language for both the front-end and the back-end. On top of that there are frameworks like React, Angular and Vue that make building large, complex web apps a lot more manageable. I've just started learning React and it's quite different from vanilla JS, but I can already see why people prefer it for bigger projects.
